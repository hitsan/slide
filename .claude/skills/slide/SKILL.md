# Slide

## Overview

「何を伝えるか」が曖昧な状態からスタートし、対話を通じて構成を固めるブレスト支援スキル。
コンテンツの詳細記入はユーザーが行う。スキルは骨格を作ることに集中する。

## 利用可能なレイアウト

このプロジェクトの `template/layouts/` に定義されたレイアウト：

| レイアウト | 適した用途 | スロット |
|---|---|---|
| `default` | テキスト・コード・一般スライド | — |
| `cover` | タイトルスライド | — |
| `two-col` | 2要素の並列比較 | `col1`, `col2` |
| `two-col-icon` | 2要素＋アイコン強調 | `col1-icon`, `col1`, `col2-icon`, `col2` |
| `cards` | 3つの要点・特徴 | `card1`, `card2`, `card3` |
| `cards-icon` | 3要点＋アイコン | `card1-icon`, `card1`, ... |
| `grid2x2` | 4つの要素 | `card1`〜`card4` |
| `grid2x2-icon` | 4要素＋アイコン | `card1-icon`, `card1`, ... |
| `list` | 縦並び3項目 | `item1`, `item2`, `item3` |
| `list-icon` | 縦並び3項目＋アイコン | `item1-icon`, `item1`, ... |
| `compare` | 2案比較＋結論 | `col1`, `col2`, `bottom` |

## Workflow

### Step 1: 基本コンテキストのヒアリング

AskUserQuestion で以下を一度に確認する：

- **トピック**: 何について話すか
- **聴衆**: 誰に向けた発表か（技術者、ビジネス層、一般など）
- **目的**: 発表後に聴衆に何をしてほしいか・何を思ってほしいか
- **発表時間**: 何分か（目安：40秒1枚）

### Step 2: キーメッセージの深掘り

ソクラテス式対話でユーザーの考えを徹底的に引き出す。質問は1回に1〜2個まで。
答えが得られたら即座に次の質問へ。全カテゴリを網羅するまで質問を続ける。

**背景・経緯（必ず聞く）:**
- 「なぜこれをやろうと思ったのか？きっかけは？」
- 「それ以前はどういう状況だったか？何が不満・不便だったか？」
- 「このテーマに取り組む前、どんな選択肢を検討したか？」

**何をしたか（必ず聞く）:**
- 「具体的に何を作った・やった・変えたか？」
- 「どんな技術・手法・ツールを使ったか？」
- 「一番苦労したところ・工夫したところは？」

**なぜ（必ず聞く）:**
- 「その方法を選んだ理由は？他の選択肢との比較は？」
- 「この設計・実装にした根拠は？」
- 「やってみて想定外だったことは？」

**結果・学び（必ず聞く）:**
- 「やった結果どうなったか？数字や変化はあるか？」
- 「聴衆に一番持ち帰ってほしいことは何か？」
- 「次にやりたいこと・残った課題は？」

**スコープ（必ず聞く）:**
- 「今回話さないこと・スコープ外は何か？」
- 「聴衆が持ちそうな疑問・反論は？どう答えるか？」

全カテゴリの回答が揃ったら **Step 2.5** へ進む。

### Step 2.5: 文脈を踏まえた追加ヒアリング

Step 2 で得た回答を踏まえ、**その内容に固有の疑問・矛盾・空白**を見つけて追加で質問する。
テンプレートの問いではなく、ユーザーの言葉から生まれた質問を投げること。

**追加質問を生成するヒント:**
- 「〇〇と言っていたが、なぜ△△ではなく□□を選んだのか？」（選択の根拠が薄い箇所）
- 「〇〇の結果、具体的に何が変わったか？数字・エピソードはあるか？」（抽象的な回答の具体化）
- 「〇〇という課題があったとのことだが、それは今も残っているか？」（未解決の糸口）
- 「聴衆の中には〇〇を知らない人もいそうだが、どこまで説明するか？」（聴衆との前提差）
- 「〇〇と〇〇の話が出たが、どちらがこの発表の主役か？」（軸がぶれている箇所）

追加質問も1〜2個ずつ。「もう十分」とユーザーが感じるまで続けてよい。
追加ヒアリングが終わったら、整理した内容をユーザーに提示して確認を得る。

### Step 3: ストーリーライン提案

目的とキーメッセージを元に、ストーリーパターンを提案する。

**パターン例:**
- **問題解決型**: 課題提示 → 原因分析 → 解決策 → 効果・まとめ
- **提案型**: 背景・文脈 → 提案内容 → 根拠・デモ → アクション
- **報告型**: エグゼクティブサマリー → 実績・数字 → 課題と対策 → 次のステップ
- **教育型**: なぜ重要か → 概念説明 → 具体例・デモ → まとめ・演習
- **比較検討型**: 現状の問題 → 選択肢A vs B → 推奨案 → 実行計画

AsciiまたはMarkdownリストでストーリーラインを提示し、変更を受け付ける。

### Step 4: スライドリスト作成

合意したストーリーラインを元に、スライド一覧を表形式で提示する。

| # | タイトル | 目的・内容 | レイアウト案 |
|---|---|---|---|
| 1 | タイトル | 発表者・発表タイトル | `cover` |
| 2 | アジェンダ | 発表の流れ | `list` |
| ... | ... | ... | ... |

レイアウト選択の基準：
- 情報の論理構造（並列・比較・リスト）に合わせる
- 「格好いいから」ではなく「情報の形に合うから」で選ぶ
- アイコン付き（`-icon`）は視覚的アクセントが必要な場合に限定

ユーザーのフィードバックを受け、追加・削除・順序変更・レイアウト変更を行い合意を得る。

### Step 5: slides.md スケルトン生成

合意したスライドリストを元に `slides.md` スケルトンを生成する。

**出力先:** `works/<topic>/slides.md`
- `<topic>` はユーザーに確認する（例: `works/lt-xxx/slides.md`）
- ディレクトリが存在しない場合は作成する

**スケルトンの形式:**

```markdown
---
theme: ../../template
title: タイトル
---

# タイトル
サブタイトル

---
layout: two-col
---

## スライドタイトル

::col1::
<!-- TODO: 左カラムの内容 -->

::col2::
<!-- TODO: 右カラムの内容 -->

---
layout: compare
---

## 比較スライドタイトル

::col1::
<!-- TODO: 案A -->

::col2::
<!-- TODO: 案B -->

::bottom::
<!-- TODO: 結論 -->
```

各スライドには `<!-- TODO: ... -->` でプレースホルダーを入れる。
ファイル生成後、`npm run dev works/<topic>/slides.md` の実行コマンドをユーザーに伝える。

## Important Notes

- 対話は短く保つ。1メッセージに質問は1〜2個まで
- Step 2 の質問は全カテゴリを必ず聞く。答えが出るたびに次を聞き続ける
- ユーザーが「わからない」「まだ決まってない」と言ったら、選択肢を提示して選ばせる
- Step を厳密に順番通りに進める必要はない。ユーザーの答えから先が見えたら飛ばしてよい
- スライド枚数の目安：**40秒1枚**（例：5分 = 7〜8枚、10分 = 15枚）。時間が決まっていれば超えないように調整する
- `cover` は最初の1枚のみ
